Видение

Коротко

Сделать создание и доводку «скиллов» для человекоподобного робота доступным через единый понятный веб-центр: от описания движения и сценария до симуляции, обучения и упаковки результата — без необходимости держать в голове весь набор консольных инструментов.

AUROSY Skill Factory в текущем репозитории — это монорепозиторий веб-клиента (SPA) этой платформы. В горизонте 1–3 лет видение — усилить роль «фабрики создателей»: команда и партнёры могут быстрее проходить цикл задумал → описал → проверил → прогнал конвейер → поставил в очередь / получил пакет, опираясь на один интерфейс и согласованные контракты с бэкендом.

Технические детали

Код SPA: каталог web/frontend/. Бэкенд (FastAPI, CLI, DDS) — отдельный репозиторий; граница — HTTP REST, WebSocket телеметрии и OpenAPI. Подробности слоя браузера: 01_frontend_architecture.md.

Философия продукта

Коротко

Инженерный инструмент, но с опорой на визуальную ясность: пользователь видит структуру задачи, статусы и логи, а не только сухие числа. Локальный «конвейер» и облачная/платформенная очередь задач явно разведены, чтобы не путать сценарии использования.

  • Визуальный центр управления. Цель UX — понятный центр (дашборд, Pose Studio, сценарии, конвейер), а не панель из одних метрик без контекста (см. продуктовый бриф в репозитории).
  • Источник истины на сервере. Валидация Phase 0 и запуск тяжёлых шагов остаются за бэкендом; браузер не ходит в DDS напрямую и не исполняет CLI.
  • Два пути к обучению. Синхронный pipeline (preprocess → playback → train) через API и асинхронная очередь Phase 5 (артефакты, jobs, пакеты) — оба первоклассные, с пояснениями в UI и справке.
  • Доступность и снижение когнитивной нагрузки. Статусы не только цветом, фокус для клавиатуры, согласованные бейджи и подсказки (дорожная карта F9–F12 в документации UI).

Бизнес-ценность платформы

Коротко

Ускоряем time-to-skill: меньше трения между ролями, прозрачные этапы и возможность масштабировать обучение и выпуск пакетов через платформенную очередь и артефакты.

  • Скорость итераций. Один интерфейс для авторинга, оценки сценария, запуска конвейера и мониторинга задач сокращает переключения между инструментами и документацией.
  • Снижение порога входа. Онбординг новых инженеров и продакт-ролей проще, когда потоки задокументированы в UI и в FAQ/справке (/help).
  • Масштабирование через Phase 5. Сохранение JSON-артефактов, постановка асинхронного train, список jobs и выпуск Skill Bundle (пакеты) поддерживают сценарии «много задач / команда / последующая дистрибуция артефактов».
  • Согласованность с контрактами API. Меньше риска «разъехавшихся» форматов между клиентом и сервисом за счёт валидации и OpenAPI на стороне бэкенда.
Технические детали

Идентификация пользователя для Phase 5: заголовок X-User-Id, настройка через VITE_PLATFORM_USER_ID и UI настроек. См. корневой README.md монорепозитория и журналы F13–F17 в 03_implementation_roadmap_frontend.md.

Ценность для пользователей

Коротко

Инженеры и продвинутые операторы получают контроль над JSON Phase 0, позой, сценарием и конвейером в одном месте, с понятной обратной связью (валидация, логи, статусы задач, телеметрия).

  • Редактирование и проверка keyframes, motion, scenario с серверной валидацией.
  • Pose Studio: визуал и управление углами суставов, согласованное с телеметрией.
  • Конструктор сценариев из mid-level действий с оценкой длительности и предупреждениями (целевое окно ~30 с).
  • Конвейер с логами preprocess / playback / train; перенос данных в форму задач платформы без «магического» автозапуска очереди.
  • Телеметрия в реальном времени по WebSocket.
  • Локализация RU/EN, справка и FAQ в приложении.

Назначение платформы

Коротко

Это единый веб-интерфейс к циклу создания и доводки скиллов и сценариев поведения роботов (в т.ч. Unitree G1 в текущем стеке): описание, визуализация, сценарии, запуск вычислений и обучения через сервер, просмотр телеметрии и (в Phase 5) работа с артефактами и пакетами.

Веб-клиент не заменяет бэкенд: он не выполняет CLI локально в браузере и не подключается к DDS напрямую — только через FastAPI (REST и WebSocket). Пути к файлам в pipeline относятся к машине, где работает бэкенд (или данные передаются в теле запроса согласно OpenAPI).

Какую проблему решаем

Коротко

Традиционно цепочка «описание движения → симуляция → обучение → артефакты» размазана по скриптам, конфигам и ручным шагам. Мы собираем ключевые шаги в согласованный UI с явными статусами и контрактами.

  • Фрагментированный toolchain. Сложно помнить порядок вызовов и где смотреть логи; конвейер в UI даёт единую точку запуска и отображения результата.
  • Сложность Phase 0. JSON-схемы и единицы измерения требуют проверки; серверная валидация и подсказки снижают число «тихих» ошибок.
  • Связка симуляции и обучения. Пользователю нужен предсказуемый путь от preprocess к train — и отдельно возможность вынести долгие задачи в очередь платформы.
  • Наблюдаемость. Телеметрия суставов в одном окне с Pose Studio упрощает сопоставление «задали / фактически».

Путь пользователя

Коротко

Старт с главной: проверка связи с API и выбор направления — авторинг и сценарии, поза, конвейер или платформа (задачи и пакеты). В конце цепочки — при необходимости телеметрия; настройки и справка доступны в любой момент.

Информационная архитектура (сайдбар)

Порядок разделов в приложении: Главная → Авторинг → Pose Studio → Сценарии → Конвейер → группа Платформа (Phase 5): Задачи, Пакеты → Телеметрия → Справка → Настройки.

Типовые маршруты

  1. Главная (/). Статус API, кратко метаданные (telemetry_mode и др.), блок «два пути к train», шорткаты в разделы.
  2. Авторинг (/authoring). Редактирование JSON, валидация.
  3. Pose Studio (/pose). Визуал и суставы; при DDS-телеметрии — предупреждение в UI (рекомендации в FAQ/справке).
  4. Сценарии (/scenarios). Каталог действий, сборка нод, оценка длительности.
  5. Конвейер (/pipeline). Preprocess, playback, train; перенос reference в задачи платформы при необходимости.
  6. Задачи (/jobs, /jobs/:jobId). Артефакты, постановка train, мониторинг.
  7. Пакеты (/packages). Skill Bundle: список, скачивание, загрузка, публикация.
  8. Телеметрия (/telemetry). Подписка на поток состояния.
  9. Справка (/help) и Настройки (/settings). FAQ, язык, базовый URL API, идентификатор платформы.
Технические детали

Маршруты заданы в web/frontend/src/App.tsx. Мост конвейер → задачи: web/frontend/src/lib/jobsPipelineBridge.ts.

Архитектура (общая)

Коротко

В браузере — React SPA с модулями по фичам; общение с одним бэкендом через REST и WebSocket. Бэкенд поднимает API, мост к CLI и поток телеметрии.

flowchart TB subgraph browser["Браузер"] routes["Маршруты"] featAuthoring["Авторинг"] featPose["Pose"] featScenario["Сценарии"] featPipeline["Конвейер"] featJobs["Задачи Phase5"] featTelem["Телеметрия"] state["Черновики"] routes --> featAuthoring routes --> featPose routes --> featScenario routes --> featPipeline routes --> featJobs routes --> featTelem featAuthoring --> state featScenario --> state end subgraph clients["Клиенты"] rest["REST"] ws["WebSocket"] end subgraph backend["FastAPI"] api["REST API"] wss["WS"] cli["CLI мост"] end featAuthoring --> rest featScenario --> rest featPipeline --> rest featJobs --> rest featTelem --> ws featPose --> rest rest --> api ws --> wss api --> cli

Браузер: модули (авторинг, поза, сценарии, конвейер, задачи, телеметрия) → REST и WebSocket → FastAPI (REST, WS, вызов CLI). Локальные черновики JSON — в браузере; валидация Phase 0 и запуск процессов — на сервере.

  • REST: единая обёртка над fetch, базовый URL из env или прокси Vite в разработке.
  • WebSocket: /ws/telemetry, по одному JSON-событию на сообщение.
  • Состояние: черновики авторинга, ноды сценария, флаги UI; персистентность на диск опциональна в будущем.

Потоки данных (кратко)

  1. Авторинг: JSON → при необходимости клиентская схема → POST /api/validate.
  2. Сценарий: список действий → сборка нод → POST /api/scenario/estimate.
  3. Конвейер: пути на хосте бэкенда или JSON в теле → subprocess → логи в UI.
  4. Платформа: сохранить артефакт → поставить train → опрос GET /api/jobs/{id} до завершения.
  5. Телеметрия: одно WS-соединение, обновление UI.
