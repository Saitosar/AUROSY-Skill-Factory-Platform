# Skill Foundry (SAI Aurosy): видение и подход

## Проблема

Пользователь задаёт движение преимущественно **кинематически**: конечные углы суставов, ключевые кадры (keyframes), сценарии из последовательности действий. Такие данные:

- не учитывают **динамику** (массы, инерцию, контакты, задержки приводов);
- не гарантируют **баланс** и устойчивость в физическом смысле;
- плохо масштабируются на «идеальное» воспроизведение на **реальном роботе** без многократной ручной доводки.

Нанимать команду для телеоперации и записи каждого навыка дорого; хранить только скрипты с PD-командами недостаточно для платформы, где тысячи пользователей создают контент.

## Решение: Robotic Skill Foundry

**Skill Foundry** — это конвейер, в котором пользовательские артефакты (позы, motion, сценарии) превращаются в **обучаемый сигнал**, проходят **симуляционное обучение** и на выходе дают **политику** (или эквивалентный артефакт), пригодный для рантайма на роботе, подключённому к платформе.

Идея не в замене редактора поз, а в **следующем слое**: из разрежённых намерений получить **плотные референсы и/или демонстрации**, затем **метрики качества** и **нейросетевой контроллер**, оптимизированный под физику и ограничения.

## Ключевой сдвиг по сравнению с «только хранением движений»

Платформа хранит не только `pose.json` и сценарии, но и:

- **плотные траектории** (референс на фиксированной частоте), полученные из keyframes и/или записи симуляции;
- **наборы демонстраций** для обучения (offline), при необходимости дополняемые online-этапом;
- **метрики** (ошибка трекинга, падения, энергия), по которым судят об «идеальности» в рамках постановки задачи;
- **экспортируемую политику** плюс **манифест** (что на входе сети, в каких единицах, с каким dt).

## Этапы конвейера (концептуально)

**1. Препроцессинг.** Ключевые кадры с временными метками преобразуются в **референсную траекторию** фиксированной частоты (например 50 Гц): интерполяция углов (сплайны; для ориентации корня при необходимости — интерполяция на SO(3), а не «ломаные» Эйлеры). При необходимости отдельно задаётся модель для **корня и контактов** (см. ограничения ниже).

**2. Среда обучения.** В физической симуляции (например MuJoCo через среду уровня mjlab/Isaac) робот получает **комбинированную задачу**: сближение с референсом (**imitation / tracking reward**), **выживание** (не падать), **штраф за энергию** и рывки. Так кинематика пользователя превращается в цель для **динамически согласованного** поведения.

**3. Обучение.** Агент (например PPO — как типичный выбор; алгоритм может быть сменяемым) обучает политику на множестве параллельных сред. На старте политика может быть нестабильной; качество растёт по мере оптимизации награды и вспомогательных метрик.

**4. Экспорт и дистрибуция.** По достижении порогов метрик в симуляции формируется **пакет**: веса (например PyTorch `.pt`), обязательный **manifest** (наблюдения, нормализация, версия модели робота, частота); при необходимости **ONNX** для внедрения без Python. Пакет публикуется в каталоге платформы; клиент с роботом загружает его и запускает через **robot runtime**, согласованный с `unitree_sdk2` или аналогом.

## Честные ограничения

- **Корень и ступни.** Если пользователь задаёт только углы суставов, **положение таза и ступней на каждом шаге** не определяется однозначно без модели контактов и выбора траектории базы. Платформа должна явно фиксировать MVP: например референс только по суставам плюс фиксированные допущения для базы, либо референс корня из отдельного правила, либо траектория, полученная из **полной сим-прокрутки** и логирования.

- **Формат на роботе.** Файл `.pt` сам по себе не описывает интерфейс с приводами; без manifest и согласованного рантайма политика непереносима. Для продакшена часто требуется **ONNX** или другой согласованный формат.

- **Sim2real.** Идеальное поведение в симе не гарантирует то же на железе без калибровки, рандомизации при обучении и/или дообучения.

- **UGC и безопасность.** Пользовательский сценарий может быть физически недостижим или опасен; платформа должна давать **статус по метрикам** (низкий скор, падения) и иметь **предохранители** на рантайме.

## Отличие от «пути производителя» (vendor stack)

Производитель робота публикует эталонные пайплайны (обучение ходьбы, трекинг motion из CSV и т.д.) под **свои** задачи и инфраструктуру. Ваш акцент — **единый продуктовый конвейер** для **краудсорсинговых** motion и сценариев: единый контракт данных, очередь обучения, каталог политик и выдача на робота по подписке к платформе. Это не замена upstream-репозиториев, а **оркестрация** и **контракты** вокруг них или собственной среды.
